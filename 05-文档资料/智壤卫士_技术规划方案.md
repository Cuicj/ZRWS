#### 2.1.1 3D 渲染技术栈详解

> **核心选型**：Three.js（WebGL 2.0），不引入 Unity WebGL / Cesium.js / Babylon.js，避免多引擎异构造成的加载体积膨胀和渲染管线冲突。

**选型对比**：

| 候选方案 | 优势 | 排除原因 |
|----------|------|----------|
| **Three.js** ✅ | Web 原生、体积小（~600KB gzip）、生态成熟、按需扩展 | — |
| Cesium.js | 地球级 GIS 可视化、3D Tiles | 体积 > 3MB gzip，本项仅需局部地形 + 点云，地球球体场景为过度设计 |
| Babylon.js | 游戏引擎级特性 | 主要面向游戏场景，Web Worker 点云解析不如 Three.js 社区方案成熟 |
| Unity WebGL | 桌面级渲染 | 导出包 > 20MB，首帧加载 > 5s，无法与 Vue 组件树深度集成 |

**Three.js 生态配套**：

| 场景 | 技术 | 版本 | 说明 |
|------|------|------|------|
| **核心引擎** | Three.js | r164+ | WebGL 2.0 渲染管线，支持 WebGPU 渐进升级 |
| **点云加载** | Potree-core / custom PCD loader | — | 支持 PCD / LAS / LAZ 格式，八叉树 LOD 分块加载 |
| **地形渲染** | Three.js PlaneGeometry + vertex displacement | — | DEM 高度图 → 3D 地形网格，纹理叠加 DOM 正射影像 |
| **模型加载** | GLTFLoader / OBJLoader / DRACOLoader | — | GLTF/GLB 优先（Draco 压缩），OBJ 兼容导入 |
| **后处理** | EffectComposer + UnrealBloomPass | — | 选点高亮、边缘检测、AO 环境光遮蔽 |
| **相机控制** | OrbitControls / FlyControls | — | 漫游/环绕/第一人称，与无人机飞行视角联动 |
| **性能分析** | stats.js + Three.js built-in renderer.info | — | 帧率/三角面数/DrawCall 实时监控 |
| **Vue 集成** | TroisJS / 自定义 Vue 组件封装 | — | Three.js 实例生命周期与 Vue 组件 `onMounted`/`onUnmounted` 绑定 |

**点云渲染流程**：

```
文件上传(MinIO)
    │
    ▼
后端 Java 解析 LAS/LAZ 头信息（点数量 / 边界 / 分类）
    │  Hutool FileUtil + 自定义二进制读取
    ▼
前端请求点云元数据（点数 / 颜色映射 / LOD 层级）
    │  REST API → Vue Store
    ▼
Three.js 异步分块加载（Web Worker 解析 XYZ+RGB）
    │  Potree-core 或自研 PCDLoader
    ▼
八叉树构建 → LOD 渲染
    │  近距：全分辨率 / 远距：降采样
    ▼
视锥裁剪（frustum culling）仅渲染可见节点
    │  配合 OrbitControls 动态更新
    ▼
渲染到 WebGL Canvas → 叠加到 Vue 组件层
```

**性能基准目标**：

| 指标 | 目标值 | 策略 |
|------|--------|------|
| 百万级点云帧率 | ≥30 FPS | 八叉树 LOD + Worker 异步解析 + GPU buffer |
| 地形瓦片加载 | ≤2s/瓦片 | DEM 预切片金字塔，Texture 渐进式 mipmap |
| 模型首帧 | ≤3s | GLB Draco 压缩 + 预加载关键模型 |
| GPU 内存 | ≤512MB | 及时 dispose geometry/texture，避免内存泄漏 |
| 浏览器兼容 | Chrome 90+ / Edge 90+ / Firefox 90+ | WebGL 2.0 最小要求 |

**Vue 集成模式**：

```typescript
// composables/useThreeScene.ts — Three.js 与 Vue 组件生命周期绑定
import * as THREE from 'three';
import { onMounted, onUnmounted, ref, type Ref } from 'vue';

export function useThreeScene(containerRef: Ref<HTMLElement | null>) {
  const renderer = ref<THREE.WebGLRenderer | null>(null);
  const scene = new THREE.Scene();
  const camera = new THREE.PerspectiveCamera(60, 1, 0.1, 10000);

  let animationId = 0;

  onMounted(() => {
    if (!containerRef.value) return;
    const r = new THREE.WebGLRenderer({ antialias: true, alpha: true });
    r.setPixelRatio(Math.min(window.devicePixelRatio, 2)); // 限制像素比防过热
    r.setSize(containerRef.value.clientWidth, containerRef.value.clientHeight);
    containerRef.value.appendChild(r.domElement);
    renderer.value = r;
    animate();
  });

  onUnmounted(() => {
    cancelAnimationFrame(animationId);
    renderer.value?.dispose();
    scene.traverse(obj => {
      if (obj instanceof THREE.Mesh) {
        obj.geometry.dispose();
        (obj.material as THREE.Material).dispose();
      }
    });
  });

  function animate() {
    animationId = requestAnimationFrame(animate);
    renderer.value?.render(scene, camera);
  }

  return { scene, camera, renderer };
}
```

```vue
<!-- ✅ Vue SFC 集成 Three.js 的块型结构 -->
<template>
  <div ref="containerRef" class="point-cloud-viewer" />
</template>

<script setup lang="ts">
import { ref, watch } from 'vue';
import { useThreeScene } from '@/composables/useThreeScene';
import { usePointCloudLoader } from '@/composables/usePointCloudLoader';

const containerRef = ref<HTMLElement | null>(null);
const { scene, camera } = useThreeScene(containerRef);

const props = defineProps<{ pointCloudUrl: string }>();

// 点云数据变更时热加载
watch(() => props.pointCloudUrl, async (url) => {
  // 先清理旧点云
  scene.children
    .filter(c => c.userData.type === 'pointCloud')
    .forEach(c => scene.remove(c));

  // 异步加载新点云
  const points = await usePointCloudLoader().load(url);
  scene.add(points);
});
</script>

  width: 100%;
  height: 60vh;
  min-height: 400px;
}
</style>
```

### 4.6 认证授权规范

> **方案**：Spring Security + JWT 无状态认证 + RBAC 权限模型。认证由 `zrws-auth` (:8100) 签发 Token，鉴权由 Gateway 全局过滤器统一拦截。

#### 4.6.1 认证流程

```
用户登录（账号+密码）
    │
    ▼
POST /api/v1/auth/login  →  zrws-auth
    │  验证账号密码（BCrypt）
    │  生成 Access Token（JWT，2h 有效期）
    │  生成 Refresh Token（随机串，7d 有效期，存 Redis）
    │  返回 { accessToken, refreshToken, userInfo }
    ▼
前端 Pinia Store 存储 Token
    │  Axios 拦截器自动带 Authorization: Bearer xxx
    ▼
Gateway AuthGlobalFilter
    │  放行白名单（/auth/**）直接路由
    │  其余请求 → JWT 验签 → 解析 userId / role / permissions
    │  写入请求头 X-User-Id / X-User-Role 透传下游
    │  过期返回 401 → 前端静默刷新或跳登录
    ▼
下游微服务
    │  从请求头获取用户信息
    │  @RequireRole 切面校验权限
```

#### 4.6.2 Token 规格

| 属性 | Access Token | Refresh Token |
|------|-------------|---------------|
| 格式 | JWT（HMAC-SHA256） | 128 位随机字符串 |
| 有效期 | 2 小时 | 7 天 |
| 存储位置 | 前端内存（Pinia） | Redis `refresh_token:{userId}` |
| 载荷 | `{ sub, userId, role, permissions, exp, iat }` | 无载荷，仅做 Redis 比对 |
| 刷新端点 | — | `POST /api/v1/auth/refresh` |
| 黑名单 | 退出时加入 Redis `blacklist:{jti}`，TTL = remaining | 删除 Redis key |

#### 4.6.3 权限模型（RBAC）

```
用户 (User) ──N:M── 角色 (Role) ──N:M── 权限 (Permission)
                                              │
                     ┌────────────────────────┬────────────────────┐
                     ▼                        ▼                    ▼
              采集权限组                 处理权限组            审批权限组
           mission:create            data:import          approval:submit
           mission:read              data:validate        approval:review
           gps:track                 data:export          approval:approve
           drone:control             3d:render            workflow:design
```

**预设角色**：

| 角色 | code | 继承 | 典型权限 |
|------|------|------|----------|
| **外业人员** | `field_staff` | — | 任务读写、飞行控制、GPS 追踪 |
| **校核员** | `checker` | field_staff | + 数据校验、质量标记 |
| **审核员** | `reviewer` | checker | + 审批流转、报告审核 |
| **审批人** | `approver` | reviewer | + 最终批准、驳回 |
| **系统管理员** | `admin` | approver | + 用户管理、角色分配、系统配置 |

#### 4.6.4 zrws-auth 模块规格

| 端点 | 方法 | 鉴权 | 说明 |
|------|------|------|------|
| `/api/v1/auth/login` | POST | 无 | 账号密码登录，返回双 Token + userInfo |
| `/api/v1/auth/refresh` | POST | Refresh Token | 刷新 Access Token，Refresh Token 不变 |
| `/api/v1/auth/logout` | POST | Access Token | Access Token 加入 Redis 黑名单，删除 Refresh Token |
| `/api/v1/auth/me` | GET | Access Token | 返回当前用户信息 + 角色 + 权限列表 |
| `/api/v1/admin/users` | CRUD | admin | 用户管理 |
| `/api/v1/admin/roles` | CRUD | admin | 角色管理 + 权限分配 |

#### 4.6.5 Gateway 过滤器

```java
// zrws-gateway/AuthGlobalFilter.java
@Component
public class AuthGlobalFilter implements GlobalFilter, Ordered {

    // 放行白名单
    private static final List<String> WHITELIST = List.of(
        "/api/v1/auth/login",
        "/api/v1/auth/refresh"
    );

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getURI().getPath();

        // 白名单直接放行
        if (WHITELIST.stream().anyMatch(path::startsWith)) {
            return chain.filter(exchange);
        }

        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (token == null || !token.startsWith("Bearer ")) {
            return unauthorized(exchange, "缺少认证令牌");
        }

        try {
            Claims claims = JwtUtil.parseToken(token.substring(7));
            ServerHttpRequest request = exchange.getRequest().mutate()
                .header("X-User-Id", claims.get("userId", String.class))
                .header("X-User-Role", claims.get("role", String.class))
                .build();
            return chain.filter(exchange.mutate().request(request).build());
        } catch (ExpiredJwtException e) {
            return unauthorized(exchange, "令牌已过期");
        } catch (JwtException e) {
            return unauthorized(exchange, "令牌无效");
        }
    }

    @Override
    public int getOrder() { return -100; }
}
```

```typescript
// stores/auth.ts — Pinia Token 管理
export const useAuthStore = defineStore('auth', () => {
  const accessToken = ref('');
  const refreshToken = ref('');
  const userInfo = ref<UserInfo | null>(null);

  async function login(username: string, password: string) {
    const res = await authApi.login({ username, password });
    accessToken.value = res.accessToken;
    refreshToken.value = res.refreshToken;
    userInfo.value = res.userInfo;
  }

  async function silentRefresh() {
    try {
      const res = await authApi.refresh(refreshToken.value);
      accessToken.value = res.accessToken;
    } catch {
      logout();
    }
  }

  function logout() {
    authApi.logout();
    accessToken.value = '';
    refreshToken.value = '';
    userInfo.value = null;
    router.push('/login');
  }

  return { accessToken, refreshToken, userInfo, login, silentRefresh, logout };
});
```

### 4.7 WebSocket 通信协议

> **选型**：SockJS + STOMP，服务端 Spring WebSocket + RabbitMQ STOMP Broker。用于无人机实时遥测、GPS 轨迹推送、AI 分析进度等需要服务端主动推送的场景。

#### 4.7.1 架构

```
前端 SockJS + STOMP.js
    │  ws://gateway/api/v1/ws
    ▼
Gateway（WebSocket 路由）
    │  鉴权：连接时携带 ?token=xxx
    │  路由：/topic/** → 广播  /queue/** → 点对点  /app/** → 服务端消费
    ▼
RabbitMQ STOMP Broker（消息中转）
    │  解耦 WebSocket 连接与业务服务
    ▼
各微服务（zrws-collect / zrws-process / zrws-analyze）
    │  通过 SimpMessagingTemplate 向 /topic/ 或 /queue/ 发送消息
```

#### 4.7.2 消息格式

所有 WebSocket 消息统一 JSON 信封：

```json

  "type": "GPS_TRACK",

  "timestamp": 1718582400000,

    "missionId": 1001,
    "lng": 116.4074,
    "lat": 39.9042,
    "alt": 120.5,
    "speed": 8.3,
```

| 心跳目标 | `/app/heartbeat` |
| 心跳载荷 | `{ "type": "HEARTBEAT", "seq": ..., "timestamp": ..., "payload": null }` |

**重连策略**（前端 SockJS）：

```typescript

  socket = new SockJS(`/api/v1/ws?token=${authStore.accessToken}`);
  stompClient = Stomp.over(socket);
  stompClient.debug = () => {}; // 关闭生产环境日志

  stompClient.connect({}, onConnect, onError);
}

function onError(): void {
  attempts++;
  if (attempts > MAX_RETRIES) {
    console.error('WebSocket 重连已达上限');
    return;
  }
  const delay = Math.min(BASE_DELAY * Math.pow(2, attempts), 30000); // 指数退避，封顶 30s
  setTimeout(connect, delay);
}
```

### 5.4 前端规范 — Vue 块型编码风格

> **强制约束**：Vue 单文件组件（`.vue`）必须采用**块型编码风格**—即 `<script>`、`<template>`、`<style>` 三块各自内部保持高内聚，块间通过 `defineProps` / `defineEmits` / `provide` / `inject` 明确声明接口，**禁止**块间隐式依赖或跨块变量引用。

**块型编码核心原则**：

| 原则 | 规则 |
|------|------|
| **块自封闭** | 每个块内部变量/函数仅在该块内可见；Template 通过插值/绑定访问 Script 暴露的值 |
| **接口显式** | Script → Template 通过 `defineProps` / `defineEmits` / `defineExpose`；组件间通过 `provide` / `inject` |
| **Style 隔离** | 默认 `<style scoped>`，全局样式放 `src/styles/` 并只在 `variables.css` 定义 Token |
| **单向数据流** | Props 只读，子组件不直接修改 props；状态变更通过 emits 通知父组件 |
| **Composable 优先** | 可复用逻辑抽离为 `useXxx.ts`，Script 块仅做组合编排 |

**Vue SFC 块型结构**：

### 5.5 变量作用域最小化

> **强制约束**：`let` / `const` / `var` 声明必须遵循"最小作用域、最小存活范围"原则。禁止 `var`，`const` 优先，`let` 仅用于必须重新赋值的场景。变量声明位置尽可能靠近首次使用的位置，杜绝函数顶部集中声明。

| 规则 | 说明 |
|------|------|
| **禁用 var** | ESLint `no-var: error`，所有变量用 `const` / `let` |
| **const 优先** | 能不变的都用 `const`；仅当语义上需要重赋值时才用 `let` |
| **块级作用域** | 变量声明放在使用它的最小块（`{}`）内，不提升到函数顶部 |
| **循环内 let** | 循环变量用 `let`，`for (let i = 0; ...)` 而非提前声明 |
| **禁止全局挂载** | 不向 `window` / `globalThis` 写属性；全局常量通过 `provide` 或 store 传递 |

#### 5.7.3 空值安全 — 禁止 null / undefined 裸显

> **强制约束**：页面上任何用户可见的文本区域，**绝对不允许**出现 `null`、`undefined`、`NaN`、`[object Object]` 等裸奔值。数据展示层必须有兜底字符串。

| 层级 | 策略 |
|------|------|
| **API 响应** | 后端必须保证字段类型一致，不允许同一字段有时是 string 有时是 null（Java `@NotNull` + VO 默认值） |
| **Pinia Store** | 从 API 取值时统一走 `??` / `||` 兜底，Store 中不存储 null |
| **Component Props** | `withDefaults()` 设置非空默认值 |
| **Template 渲染** | 禁止 `{{ someNullable }}`；必须 `{{ someNullable ?? '-' }}` 或自定义 `v-safe-text` 指令 |
| **可选链** | 访问深层属性用 `?.` ，末尾跟上 `??` 兜底 |

```typescript
// ✅ 自定义空值安全指令
// directives/vSafeText.ts
import type { Directive } from 'vue';

export const vSafeText: Directive<HTMLElement, unknown> = {
  mounted(el, binding) {
    updateText(el, binding.value);
  },
  updated(el, binding) {
    updateText(el, binding.value);
  },
};

function updateText(el: HTMLElement, value: unknown) {
  const fallback = el.dataset.safeFallback ?? '-';
  if (value === null || value === undefined || (typeof value === 'number' && isNaN(value))) {
    el.textContent = fallback;
  } else if (typeof value === 'object') {
    el.textContent = fallback;  // 防止 [object Object]
  } else {
    el.textContent = String(value);
  }
}
```

```vue
<!-- ✅ 使用指令 -->
<td v-safe-text="mission.soilType"></td>          <!-- null / undefined → "-" -->
<td v-safe-text="mission.area" data-safe-fallback="待测算"></td>

<!-- ✅ 模板表达式兜底 -->

<span>{{ waypoints.length ? waypoints.length + '个' : '—' }}</span>
<span>{{ formatPercent(coverage) || '暂无数据' }}</span>
```

#### 5.9.4 敏感信息脱敏

> **强制规则**：生产环境禁止在任何日志（包括 DEBUG）中打印密码、Token、手机号、身份证号等敏感字段。

```java
// zrws-common/log/SensitiveLogConverter.java
// Logback 自定义转换器，对敏感字段自动掩码
public class SensitiveLogConverter extends MessageConverter {
    // 掩码规则
    private static final Pattern PHONE   = Pattern.compile("(1[3-9]\\d)\\d{4}(\\d{4})");
    private static final Pattern ID_CARD = Pattern.compile("(\\d{3})\\d{11}(\\d{4})");
    private static final Pattern TOKEN   = Pattern.compile("(Bearer\\s)([A-Za-z0-9._-]+)");

    @Override
    public String convert(ILoggingEvent event) {
        String msg = super.convert(event);
        msg = PHONE.matcher(msg).replaceAll("$1****$2");     // 138****5678
        msg = ID_CARD.matcher(msg).replaceAll("$1***********$2"); // 110***********1234
        msg = TOKEN.matcher(msg).replaceAll("$1<MASKED>");   // Bearer <MASKED>
        return msg;
    }
}
```

脱敏清单：

| 字段类型 | 脱敏规则 | 示例 |
|----------|----------|------|
| 手机号 | 保留前3后4，中间4位 `*` | `138****5678` |
| 身份证号 | 保留前3后4，中间11位 `*` | `110***********1234` |
| Access Token / Refresh Token | 完全替换为 `<MASKED>` | `<MASKED>` |
| 密码字段（password/pwd/passwd） | 完全替换为 `***` | `***` |
| 邮箱 | 保留首字符 + 域名 | `1***@qq.com` |

#### 5.9.5 访问审计 — 黑白名单

**白名单**：必须全量记录审计日志的接口（安全敏感操作）：

```java
// zrws-gateway/filter/AccessLogFilter.java
@Component
public class AccessLogFilter implements GlobalFilter, Ordered {

    // 白名单：必须记录审计日志的路径
    private static final Set<String> AUDIT_WHITELIST = Set.of(
        "/api/v1/auth/login",
        "/api/v1/auth/logout",
        "/api/v1/admin/users",         // 用户管理
        "/api/v1/admin/roles",         // 角色管理
        "/api/v1/approval/*/approve",  // 审批操作
        "/api/v1/mission/*/delete"     // 任务删除
    );

    // 黑名单：禁止记录请求体的路径（含敏感数据）
    private static final Set<String> BODY_BLACKLIST = Set.of(
        "/api/v1/auth/login",        // 含密码
        "/api/v1/auth/refresh"       // 含 Refresh Token
    );

    private final AccessLogService accessLogService;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getURI().getPath();

        // 仅白名单全量审计
        if (AUDIT_WHITELIST.stream().anyMatch(path::matches)) {
            String userId = exchange.getRequest().getHeaders().getFirst("X-User-Id");
            String ip = exchange.getRequest().getRemoteAddress().getAddress().getHostAddress();
            String method = exchange.getRequest().getMethod().name();
            String params = extractParams(exchange, path);

            accessLogService.log(userId, ip, method, path, params);
        }

        return chain.filter(exchange);
    }

    // 请求体在黑名单中 → 不记录
    private String extractParams(ServerWebExchange exchange, String path) {
        if (BODY_BLACKLIST.stream().anyMatch(path::matches)) {
            return "<MASKED>";
        }
        return exchange.getAttribute("cachedRequestBody"); // 配合 @CachedBody 过滤器
    }
}
```

```typescript
    const status = error.response?.status;

    if (status === 401) {
      useUserStore().logout();         // Token 过期 → 跳登录
      ElMessageBox.alert('登录已过期，请重新登录');
    } else if (status === 403) {
      ElMessage.error('无操作权限');
    } else if (status >= 500) {
      ElMessage.error('服务器繁忙，请稍后重试');
    }

    return Promise.reject(error);
  }
);
```

#### 5.10.7 异常处理约束

| 约束 | 说明 |
|------|------|
| Controller 禁 try-catch | 全部交给 `GlobalExceptionHandler` |
| Service 只抛 BizException | 禁止 `throw new RuntimeException("xxx")` |
| 错误码必须枚举 | 禁止硬编码 `R.fail(1234, "某个错误")` |
| 生产不输出 stacktrace | 仅 `log.error` 记录完整堆栈，响应体只含错误码和 message |
| 第三方调用必须 try | Feign / MQ / 外部 API 必须捕获并转为 BizException |
| traceId 必带 | 响应体 `traceId` 由全局拦截器统一注入，支持日志串联 |

```java
@Service
public class CoordConfigService {

            FROM region_coord_config
            WHERE ST_Contains(boundary, ST_GeomFromText(CONCAT('POINT(', ?, ' ', ?, ')'), 4326))
            LIMIT 1
```

### 8.5 限流与熔断

> **方案**：Sentinel 1.8.x，通过 Nacos 持久化规则，控制台实时修改推送。覆盖 Gateway 入口限流 + 微服务调用熔断 + 热点参数限流三层。

#### 8.5.1 Sentinel 架构

```
Nacos 持久化规则（启动加载）
    │
    ▼
Sentinel Dashboard（控制台 :8080）
    │  实时监控 / 规则推送
    ▼
┌─────────────────────────────────────────────────────┐
│ Gateway 层（入口限流）                                 │
│   UrlCleaner：清洗 URL 变量 /api/v1/mission/{id}       │

┌─────────────────────────────────────────────────────┐
│ 微服务层（调用熔断 + 热点限流）                         │
```

### 8.7 缓存策略规范

> **方案**：Caffeine（L1 本地缓存）+ Redis + Redisson（L2 分布式缓存）+ 分布式锁 / 布隆过滤器。

#### 8.7.1 缓存分层

```
请求
  │
  ▼
L1 Caffeine（JVM 堆内，10min TTL，最多 5000 条）
  │  命中 → 直接返回
  │  未命中
  ▼
L2 Redis（分布式，按业务设 TTL）
  │  命中 → 回填 L1 + 返回
  │  未命中
  ▼
MySQL
  │  查询结果 → 写入 L2 → 写入 L1 → 返回
```

| 层级 | 技术 | 容量 | TTL | 用途 |
|------|------|------|-----|------|
| L1 | Caffeine | 5000 条 | 10 min | 热点配置、土壤分类、坐标系参数 |
| L2 | Redis | 内存限制 | 按业务 | 分布式共享缓存、Session、分布式锁 |
| 持久层 | MySQL | — | — | 最终数据源 |

#### 8.7.2 Key 命名规范

```
{project}:{module}:{entity}:{identifier}

示例：
  zrws:sys:coords:WGS84          # 坐标系配置
  zrws:sys:soil_type:SAND         # 土壤分类
  zrws:mission:detail:1001        # 任务详情
  zrws:auth:token:refresh:10086   # 用户 Refresh Token
  zrws:lock:mission:1001          # 分布式锁
  zrws:counter:mission:daily      # 每日任务计数器
```

| 分段 | 说明 | 示例 |
|------|------|------|
| `zrws` | 项目前缀，避免与其他系统冲突 | 固定 |
| `module` | 模块：sys / mission / auth / ai / process | 按微服务划分 |
| `entity` | 实体：coords / soil_type / detail / token / lock | 实体名 |
| `identifier` | 唯一标识：code / id / daily | 业务主键 |

#### 8.7.3 TTL 策略

| 数据类型 | TTL | 淘汰策略 | 说明 |
|----------|-----|----------|------|
| 系统配置（坐标系/土壤分类） | 1 小时 | 主动删除 | Nacos 配置变更时清除对应 Key |
| 任务详情 | 5 分钟 | 主动删除 + TTL | 任务状态变更时清除 |
| 审批流模板 | 30 分钟 | TTL | 模板变更低频 |
| Refresh Token | 7 天 | TTL | 与 Token 有效期一致 |
| 分布式锁 | 30 秒 | TTL | 防止死锁，必须设超时 |
| 接口幂等 Key | 10 分钟 | TTL | 防止重复提交 |
| 限流计数器 | 滑动窗口 | TTL | Sentinel 兜底 |

#### 8.7.4 防穿透、击穿、雪崩

| 问题 | 方案 | 实现 |
|------|------|------|
| **缓存穿透**（查不存在的数据） | 布隆过滤器 + 空值缓存 | Redisson RBloomFilter 预热所有合法 ID；查询不存在则缓存空值 `NULL::30s` |
| **缓存击穿**（热点 Key 过期瞬间高并发） | 分布式锁 + 互斥更新 | `Redisson.getLock("lock:" + key)` 抢锁，抢到查 DB 写缓存，未抢到自旋等待 |
| **缓存雪崩**（大批 Key 同时过期） | TTL 加随机偏移 | 基础 TTL ± 随机 300s，避免集中过期 |

```java
// 防击穿：分布式锁互斥更新
public MissionDetail getMissionDetail(Long missionId) {
    String key = "zrws:mission:detail:" + missionId;
    String lockKey = "zrws:lock:mission:" + missionId;

    // L1
    MissionDetail cached = caffeineCache.getIfPresent(key);
    if (cached != null) return cached;

    // L2
    cached = (MissionDetail) redisTemplate.opsForValue().get(key);
    if (cached != null) {
        caffeineCache.put(key, cached);
        return cached;
    }

    // 分布式锁互斥更新
    RLock lock = redissonClient.getLock(lockKey);
    try {
        if (lock.tryLock(3, 30, TimeUnit.SECONDS)) {
            // 双重检查
            cached = (MissionDetail) redisTemplate.opsForValue().get(key);
            if (cached != null) return cached;

            // 查 DB
            MissionDetail detail = missionMapper.selectById(missionId);
            if (detail != null) {
                int ttl = 300 + ThreadLocalRandom.current().nextInt(300); // TTL 加随机偏移
                redisTemplate.opsForValue().set(key, detail, ttl, TimeUnit.SECONDS);
                caffeineCache.put(key, detail);
                return detail;
            } else {
                // 缓存空值 30s 防穿透
                redisTemplate.opsForValue().set(key, "NULL", 30, TimeUnit.SECONDS);
                return null;
            }
        }
    } finally {
        if (lock.isHeldByCurrentThread()) lock.unlock();
    }
    return null;
}
```

#### 8.7.5 缓存更新策略

| 场景 | 策略 | 说明 |
|------|------|------|
| 任务创建 | 不缓存（写入后首次查询才缓存） | 延迟加载 |
| 任务状态变更 | Cache-Aside：先更新 DB → 后删除缓存 Key | 下次查询时重新加载 |
| 系统配置变更（Nacos） | Nacos Listener → 清除 Redis + Caffeine | 配置热更新 |
| 批量数据导入 | 操作完成后清除相关前缀 Key | `redisTemplate.delete(redisTemplate.keys("zrws:mission:*"))` |
| 定时刷新热点 | `@Scheduled` 每 10 分钟刷热点 Key | 避免过期瞬间集中查 DB |

```java
// Redisson 分布式锁工具
@Component
public class LockUtil {
    private final RedissonClient redissonClient;

    public <T> T tryLockAndExecute(String key, long waitSeconds, long leaseSeconds,
                                    Supplier<T> action, Supplier<T> fallback) {
        RLock lock = redissonClient.getLock("zrws:lock:" + key);
        try {
            if (lock.tryLock(waitSeconds, leaseSeconds, TimeUnit.SECONDS)) {
                return action.get();
            }
            return fallback.get();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return fallback.get();
        } finally {
            if (lock.isHeldByCurrentThread()) lock.unlock();
        }
    }
}
```

```java
public class MqProducer {

    public void send(String exchange, String routingKey, Object payload) {
        MqMessage message = MqMessage.builder()
```

#### 8.9.3 消息消费规范

```java
// zrws-process/.../MissionEventListener.java
@Component
public class MissionEventListener {

    @RabbitListener(
        queues = "mission.process.queue",           // 主队列
        concurrency = "3-5",                        // 3~5 个消费者动态扩缩
        ackMode = "MANUAL"                          // 手动确认
    )
    public void onMissionEvent(
        @Payload MqMessage message,
        Message rawMessage,
        Channel channel,
        @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag
    ) throws IOException {

        try {
            // 幂等性校验 → 防止重复消费
            if (idempotentService.hasProcessed(message.getMessageId())) {
                channel.basicAck(deliveryTag, false);
                return;
            }

            // 业务处理
            MissionChangedEvent event = JSONUtil.toBean(message.getPayload(), MissionChangedEvent.class);
            processService.handleMissionChanged(event);

            // 幂等标记
            idempotentService.markProcessed(message.getMessageId());

            // 手动 ACK
            channel.basicAck(deliveryTag, false);

        } catch (Exception e) {
            // 进入重试流程
            channel.basicNack(deliveryTag, false, false); // 不重新入队，走 DLX
        }
    }
}
```

#### 8.9.4 消息重试 — 死信队列 + 指数退避

```
主交换机 (mission.event)
      │
      ▼
主队列 (mission.process.queue) ── 消费失败 ──► 死信交换机 (mission.process.dlx)
      :                               :                │
      : x-dead-letter-exchange        :                ▼
      : x-dead-letter-routing-key      :         死信队列 (mission.process.dlq)
      :                               :
      └── 成功 ──► ACK                └── 重试 3 次后 Route 到 DLQ
```

```java
// RabbitMQ 配置：声明主队列 + DLX + DLQ
@Configuration
public class MqRetryConfig {

    @Bean
    public Queue processQueue() {
        return QueueBuilder.durable("mission.process.queue")
            .deadLetterExchange("mission.process.dlx")      // 失败转 DLX
            .deadLetterRoutingKey("mission.process.dlq")    // DLX 路由键
            .ttl(60_000)                                     // 定时重试窗口 60s
            .maxLength(10_000)                               // 队列长度限制
            .build();
    }

    @Bean
    public Queue processDlq() {
        return QueueBuilder.durable("mission.process.dlq")
            .deadLetterExchange("mission.process.exchange") // 扔回原交换机
            .deadLetterRoutingKey("mission.status.changed")
            .ttl(getRetryDelay(2))                          // 第 2 次延迟 30s
            .build();
    }

    /**
     * 指数退避：retryCount=1 → 10s, 2 → 30s, 3 → 90s, 超过直接丢弃
     */
    private int getRetryDelay(int retryCount) {
        return switch (retryCount) {
            case 1  -> 10_000;
            case 2  -> 30_000;
            case 3  -> 90_000;
            default -> throw new IllegalStateException("超过最大重试次数");
        };
    }
}
```

消费端增量重试计数器：

```java
// DLQ 消息解析重发
@RabbitListener(queues = "mission.process.dlq")
public void retryFromDlq(MqMessage message, Message rawMessage, Channel channel, long deliveryTag)
    throws IOException {

    int retryCount = message.getRetryCount() + 1;

    if (retryCount > 3) {
        // 超过 3 次 → 移入最终死信 + 写入补偿表 + 告警
        compensationService.record(message, "超过最大重试次数");
        channel.basicAck(deliveryTag, false);
        return;
    }

    // 更新重试次数，重新投递
    message.setRetryCount(retryCount);
    rabbitTemplate.convertAndSend("mission.event", message.getType(), message);
    channel.basicAck(deliveryTag, false);
}
```

重试策略汇总：

| 重试次数 | 延迟 | 来源 | 失败后 |
|----------|------|------|--------|
| 第 1 次 | 10s | 主队列 TTL → DLX → DLQ | 进入 DLQ → 第 2 次重试 |
| 第 2 次 | 30s | DLQ TTL → DLX → 原队列 | 进入 DLQ → 第 3 次重试 |
| 第 3 次 | 90s | DLQ TTL → DLX → 原队列 | 进入最终死信 → 写入补偿表 + 告警 |

#### 8.9.5 消息异常补偿

**补偿表**：消费失败超过 3 次的消息写入 DB，由定时任务兜底重试。

```sql
CREATE TABLE t_mq_compensation (
    id              BIGINT       NOT NULL,
    message_id      VARCHAR(64)  NOT NULL COMMENT 'MQ 消息 ID',
    exchange        VARCHAR(128) NOT NULL,
    routing_key     VARCHAR(128) NOT NULL,
    payload         MEDIUMTEXT   NOT NULL COMMENT '消息体 JSON',
    fail_reason     TEXT         COMMENT '失败原因',
    retry_count     INT          NOT NULL DEFAULT 0,
    max_retry       INT          NOT NULL DEFAULT 10,
    next_retry_at   DATETIME     NOT NULL COMMENT '下次重试时间',
    status          TINYINT      NOT NULL DEFAULT 0 COMMENT '0-待重试 1-成功 2-人工处理',
    create_time     DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time     DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    UNIQUE KEY uk_message_id (message_id),
    INDEX idx_status_next_retry (status, next_retry_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='MQ 消息异常补偿表';
```

```java
// zrws-common/mq/CompensationService.java
@Component
public class CompensationService {

    private final MqCompensationMapper mapper;
    private final RabbitTemplate rabbitTemplate;

    /** 写入补偿表 */
    public void record(MqMessage message, String failReason) {
        MqCompensation c = new MqCompensation();
        c.setMessageId(message.getMessageId());
        c.setExchange(getExchangeByType(message.getType()));
        c.setRoutingKey(message.getType());
        c.setPayload(message.getPayload());
        c.setFailReason(failReason);
        c.setNextRetryAt(LocalDateTime.now().plusMinutes(5)); // 5 分钟后首次重试
        mapper.insert(c);
    }

    /** 定时任务：每 30 秒扫描补偿表 */
    @Scheduled(fixedDelay = 30_000)
    public void scheduledRetry() {
        List<MqCompensation> list = mapper.selectList(
            new LambdaQueryWrapper<MqCompensation>()
                .eq(MqCompensation::getStatus, 0)
                .le(MqCompensation::getNextRetryAt, LocalDateTime.now())
                .last("LIMIT 50")                  // 每次最多 50 条
        );

        for (MqCompensation c : list) {
            try {
                rabbitTemplate.convertAndSend(c.getExchange(), c.getRoutingKey(), c.getPayload());
                c.setStatus(1);  // 标记成功
            } catch (Exception e) {
                c.setRetryCount(c.getRetryCount() + 1);
                if (c.getRetryCount() >= c.getMaxRetry()) {
                    c.setStatus(2); // 人工处理
                    log.error("补偿重试耗尽 messageId={}", c.getMessageId());
                } else {
                    // 指数退避：5min → 10min → 20min → 40min ...
                    long delay = (long) Math.pow(2, c.getRetryCount()) * 5;
                    c.setNextRetryAt(LocalDateTime.now().plusMinutes(delay));
                }
            }
            mapper.updateById(c);
        }
    }
}
```

补偿重试时间线：

| 补偿次数 | 间隔 | 累计时间 |
|----------|------|----------|
| 第 1 次 | 5min | 5min |
| 第 2 次 | 10min | 15min |
| 第 3 次 | 20min | 35min |
| 第 4 次 | 40min | 1h15min |
| ... | ... | ... |
| 第 10 次 | — | 标记为「人工处理」 |

#### 8.9.6 分布式事务 — Saga 编排

> **定位**：长事务场景（如"采集完成 → 触发内业处理 → AI 分析 → 审批"），采用 Saga Choreography（编排式），通过事件驱动串联各服务本地事务。

```
采集服务                    内业处理服务                AI 分析服务                审批服务
  │                           │                          │                         │
  │ (1) 本地事务：任务状态=已完成   │                          │                         │
  │---- success                │                          │                         │
  ├──┬──► MISSION_COMPLETED    │                          │                         │
  │  │                     (2) 本地事务：处理记录创建         │                         │
  │  │                     │---- success                │                         │
  │  │                     ├──┬──► PROCESS_COMPLETED     │                         │
  │  │                     │  │                      (3) 本地事务：AI 任务创建        │
  │  │                     │  │                      │---- success                │
  │  │                     │  │                      ├──┬──► ANALYSIS_COMPLETED   │
  │  │                     │  │                      │  │                      (4) 本地事务：审批单创建
  │  │                     │  │                      │  │                      │---- success
  │  │                     │  │                      │  │                      │
  │  │                     │  │                      │  │ 🔴 (5) 审批驳回 → 触发级联回滚
  │  │◄── 补偿：任务回退     │◄── 补偿：处理记录废弃     │◄── 补偿：AI结果标记无效      │
```

**回滚规则**：每一步的补偿操作由各服务本地定义，通过 `ROLLBACK` 事件驱动：

| 阶段 | 保障措施 | 说明 |

```yaml
metadata:
  name: zrws-gateway
  namespace: zrws-prod
spec:
  replicas: 2
  strategy:
    type: RollingUpdate

      maxSurge: 1

      app: zrws-gateway

        - name: gateway
          image: registry.zrws.com/zrws-gateway:{{ .Values.imageTag }}
          ports:
            - containerPort: 8080
          resources:
            requests: { cpu: "500m", memory: "512Mi" }

            httpGet: { path: /actuator/health/liveness, port: 8080 }
            initialDelaySeconds: 30
            periodSeconds: 10

            httpGet: { path: /actuator/health/readiness, port: 8080 }

            periodSeconds: 5
          env:
```

#### 8.11.3 核心业务指标

```java
// 自定义指标注册
@Component
public class MissionMetrics {

    private final Counter missionCreatedCounter;
    private final Timer missionCreateTimer;
    private final Gauge activeMissionGauge;

    public MissionMetrics(MeterRegistry registry) {
        this.missionCreatedCounter = Counter.builder("zrws.mission.created.total")
            .tag("module", "collect")
            .description("采集任务创建总数")
            .register(registry);

        this.missionCreateTimer = Timer.builder("zrws.mission.create.duration")
            .description("任务创建耗时")
            .publishPercentiles(0.5, 0.95, 0.99)    // P50/P95/P99
            .register(registry);

        this.activeMissionGauge = Gauge.builder("zrws.mission.active.count",
                () -> missionService.countByStatus(MissionStatus.ACTIVE))
            .description("当前进行中的采集任务数")
            .register(registry);
    }

    public void recordCreate(long durationMs) {
        missionCreatedCounter.increment();
        missionCreateTimer.record(durationMs, TimeUnit.MILLISECONDS);
    }
}
```

#### 8.11.5 告警规则

```yaml
# alert.rules.yml
groups:

  # ──────────── P0 紧急 ────────────
  - name: p0_critical
    rules:
      - alert: ServiceDown
        expr: up{job=~"zrws-.*"} == 0
        for: 1m
        labels: { severity: critical, level: P0 }
        annotations:
          summary: "{{ $labels.job }} 服务挂载"
          description: "实例 {{ $labels.instance }} 已失联超过 1 分钟"

      - alert: GatewayErrorRate
        expr: rate(http_server_requests_seconds_count{job="zrws-gateway",status=~"5.."}[5m]) > 0.01
        for: 2m
        labels: { severity: critical, level: P0 }
        annotations:
          summary: "Gateway 5xx 错误率 > 1%"

  # ──────────── P1 严重 ────────────
  - name: p1_warning
    rules:
      - alert: HighMemoryUsage
        expr: jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} > 0.85
        for: 5m
        labels: { severity: warning, level: P1 }
        annotations:
          summary: "{{ $labels.application }} JVM 堆内存使用率 > 85%"

      - alert: DbConnectionPoolExhausted
        expr: hikaricp_connections_active / hikaricp_connections_max > 0.9
        for: 3m
        labels: { severity: warning, level: P1 }
        annotations:
          summary: "数据库连接池使用率 > 90%"

      - alert: HighGcPause
        expr: increase(jvm_gc_pause_seconds_sum[1m]) > 10
        for: 2m
        labels: { severity: warning, level: P1 }
        annotations:
          summary: "GC 暂停累计 > 10s/min，可能即将 OOM"

      - alert: MqQueueBacklog
        expr: rabbitmq_queue_messages{queue=~"mission.*|processing.*"} > 500
        for: 5m
        labels: { severity: warning, level: P1 }
        annotations:
          summary: "MQ 队列 {{ $labels.queue }} 积压 > 500 条"

  # ──────────── P2 关注 ────────────
  - name: p2_info
    rules:
      - alert: SlowApi
        expr: histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m])) > 3
        for: 10m
        labels: { severity: info, level: P2 }
        annotations:
          summary: "{{ $labels.uri }} P99 响应 > 3s"

      - alert: AiInferenceSlow
        expr: histogram_quantile(0.95, rate(zrws_ai_inference_duration_seconds_bucket[5m])) > 30
        for: 10m
        labels: { severity: info, level: P2 }
        annotations:
          summary: "AI 推理 P95 > 30s"

      - alert: CacheHitRateLow
        expr: rate(zrws_cache_hit_total[5m]) / rate(zrws_cache_access_total[5m]) < 0.5
        for: 15m
        labels: { severity: info, level: P2 }
        annotations:
          summary: "缓存命中率 < 50%"
```

### 8.12 容灾与备份

> **原则**：RTO ≤ 30min，RPO ≤ 5min。数据层多副本 + 定时全量备份 + 实时增量日志，应用层 K8s 自愈 + 多副本无状态。

#### 8.12.1 目标定义

| 指标 | 目标值 | 说明 |
|------|--------|------|
| RTO（恢复时间） | ≤ 30min | 从故障发生到服务恢复 |
| RPO（数据丢失） | ≤ 5min | 可容忍的最大数据丢失窗口 |
| 全年可用性 | 99.9%（3 个 9） | 年停机 ≤ 8.76h |

#### 8.12.4 数据恢复流程


#### 8.12.5 应用层高可用

| 组件 | 高可用方案 | 故障转移 |
|------|-----------|----------|
| Gateway | K8s Deployment replicas: 2 + PodAntiAffinity | 自动漂移到健康节点 |
| 微服务 | K8s Deployment replicas: 2 + HPA (min:2, max:6) | 自动漂移 + 自动扩缩容 |
| MySQL | Group Replication 3 节点（单主） | 主故障 → 自动选举新主 |
| Redis | Cluster 6 节点（3 主 3 从） | 主故障 → 从自动提升 |
| RabbitMQ | Cluster 3 节点 + HA Policy（all） | 节点故障 → 镜像队列接管 |
| Nacos | Cluster 3 节点 + 内置 Raft | Leader 故障 → 自动选举 |
| MinIO | Distributed 8 节点（纠删码 EC 4+4） | 最多允许 4 节点故障 |

> - Mock 数据必须与 `@Schema(example=...)` 声明保持一致。

**8.13.8.3 复杂度判定与 PUML 触发规则**

| 复杂度等级 | 判定标准 | PUML 要求 | 示例功能 |
|-----------|---------|----------|---------|
| O(1) | 单表 CRUD，无嵌套 | 不要求 | 查询任务列表、获取任务详情 |
| O(n) | 单表遍历 + 简单过滤 | 不要求 | 批量删除任务、导出 CSV |
| O(n²) | 双表 JOIN 或双层嵌套遍历 | 不要求 | 关联设备查询、权限校验 |
| **O(n³)** | **三层嵌套 / 多表关联 + 业务状态机 / AI 推理 / 渲染管线** | **必须 PUML** | GPS 实时追踪、点云渲染、土质 AI 分类、审批工作流 |
| O(n³+) | 图算法 / 路径规划 / 多点云融合 / 分布式事务编排 | 必须 PUML + 时序图 | 区域最优路径规划、多源点云融合、Saga 回滚流程 |

> **判定流程**：TL 在需求评审时标注复杂度，O(n³) 及以上功能若无 `.puml` 流程图，**评审不通过**。

**8.13.8.4 PUML 流程图绘制标准**

每个 `.puml` 文件必须满足：

```
@startuml
title 功能名称（复杂度：O(n³)）

actor 用户 as user
participant "前端" as front
participant "Gateway" as gw
participant "微服务" as svc
participant "AI 服务" as ai   // 如适用
database "MySQL" as db
database "Redis" as cache
queue "RabbitMQ" as mq         // 如适用

== 正常流程 ==
user -> front: 触发操作
front -> gw: HTTP 请求
gw -> gw: Token 校验 + 限流
gw -> svc: 转发请求
svc -> db: [SQL] SELECT ...
svc -> cache: [Redis] GET ...
svc -> ai: [gRPC] 调用 AI 模型
ai --> svc: 推理结果
svc -> db: [SQL] INSERT ...
svc -> mq: 发送 domain.event
svc --> gw: R<T> 响应
gw --> front: HTTP 200
front --> user: UI 更新

== 异常流程 ==
alt 参数校验失败
    svc --> gw: 400 + 错误码
    gw --> front: 400
    front --> user: 错误提示
else AI 服务超时
    ai -[#red]-> svc: 5s 超时
    svc -> svc: Sentinel 熔断降级
    svc --> gw: 503 + FALLBACK
    gw --> front: 503
    front --> user: 系统繁忙，请稍后重试
else 数据库死锁
    svc -> db: [SQL] 重试 3 次
    svc --> gw: 500 + 内部错误
    gw --> front: 500
    front --> user: 操作失败，已记录日志
end

@enduml
```

**必需标注的节点类型**：

| 标注 | 格式 | 示例 |
|------|------|------|
| SQL 操作 | `[SQL] SELECT/INSERT/UPDATE/DELETE 表名` | `svc -> db: [SQL] INSERT t_mission` |
| 缓存操作 | `[Redis] GET/SET/DEL key` | `svc -> cache: [Redis] SET zrws:mission:1024` |
| MQ 消息 | `发送/消费 Topic.事件` | `svc -> mq: 发送 mission.created` |
| RPC 调用 | `[gRPC/Feign] 方法名` | `svc -> ai: [gRPC] classifySoil` |
| 异常路径 | `alt/else` 块，红色箭头 | `svc -[#red]-> gw: 500` |

| 层级 | 输入定义深度 | 输出定义深度 | 载体 |

```

```

### 二、测绘地理信息安全保密要点

根据《测绘地理信息管理工作国家秘密范围的规定》(自然资发〔2020〕95号)：

**绝密级事项**（泄露后对国家安全造成特别严重威胁）：
- 国家大地坐标系、地心坐标系之间的转换参数(精度优于±1m)
- 1:1万及以上分辨率基础地理信息覆盖全国范围的连续数据

**机密级事项**（泄露后对国家安全造成严重威胁）：
- 1:2.5万、1:5万、1:10万基础地理信息数据
- 军事禁区以外但涉及国家安全的测绘成果
- 精度优于±5m的数字高程模型(DEM)覆盖重点区域

**秘密级事项**（泄露后对国家安全造成威胁）：
- 涉及国家安全的测绘地理信息核心技术水平
- 1:25万、1:50万基础地理信息数据
- 分辨率优于0.5m的正射影像覆盖敏感区域

**本项目安全合规措施**：
- 所有坐标系参数存储均做加密处理
- DEM/DSM数据使用AES-256加密存储
- 数据导出前自动检测涉密区域，触发审批流程
- 用户权限基于RBAC分级，涉密操作需双因子认证

### 四、坐标系统——CGCS2000 vs WGS84 对比

#### 4.1 核心参数对比

| 特性 | CGCS2000（中国法定坐标系） | WGS84（国际GPS标准） |
|------|---------------------------|---------------------|
| 原点类型 | 地心坐标系 | 地心坐标系 |
| 坐标原点 | 地球质量中心 | 地球质量中心 |
| 参考椭球 | GRS 1980 椭球 | WGS 84 椭球 |
| 长半轴(a) | 6,378,137.0 m | 6,378,137.0 m |
| 扁率倒数(1/f) | 298.257222101 | 298.257223563 |
| 参考框架 | ITRF97 | ITRF2008/ITRF2014/ITRF2020（随版本更新） |
| 框架历元 | 2000.0（固定历元） | 观测历元（动态，无固定历元） |
| 法律地位 | 中国法定坐标系，全部测绘成果强制使用 | 国际通用，GPS/GNSS原始输出 |
| 应用场景 | 国土测绘、工程放样、不动产登记、政府GIS | 手机导航、车载GPS、国际航线 |

#### 4.2 差异说明

| 差异维度 | 具体差异 | 量级 |
|---------|---------|------|
| 椭球扁率差异 | CGCS2000 f=1/298.257222101，WGS84 f=1/298.257223563 | 大地纬度差异最大约0.0001″，平面约1-3cm |
| 历元位移差异 | 归因于欧亚板块每年约3-4cm东向+北向运动 | 到2025年累计位移约0.6-1.0m |
| 兼容性 | 2000.0历元归算后两者相容 | ≤1cm（2000.0历元）/ 分米级（当前历元） |

#### 4.3 本项目坐标策略

```
┌────────────────────────────────────────────────────────┐
│  本项目统一采用 CGCS2000 坐标系，SRID 4326            │
│                                                        │
│  外部GPS/WGS84数据 → 七参数转换 → CGCS2000存储        │
│  输出给第三方 → 按需转换(七参数/格网改正)               │
│  数据库：GEOMETRY类型 + R-Tree空间索引                │
└────────────────────────────────────────────────────────┘
```

**转换方法**：
- **布尔莎七参数模型**：3平移(ΔX,ΔY,ΔZ) + 3旋转(Rx,Ry,Rz) + 1尺度(K)
- **格网改正模型**：国家发布的高精度格网转换文件(精度可达cm级)
- 转换公式：`[X,Y,Z]_CGCS2000 = (1+K)·R·[X,Y,Z]_WGS84 + [ΔX,ΔY,ΔZ]`

#### 4.4 高程基准

| 基准 | 说明 |
|------|------|
| 1985国家高程基准 | 中国法定高程基准，本项目高程系统默认 |
| WGS84椭球高 | GPS直接输出，为大地高(椭球高)，非正常高 |
| 转换关系 | 正常高 = 大地高 - 高程异常(ζ)，通过大地水准面模型计算 |

### 六、植被指数(NDVI)标准——GB/T 30115 vs 国际标准

#### 6.1 NDVI计算公式

**标准定义**（GB/T 30115-2013，引用自GB/T 14950-2009）：

$$NDVI = \frac{\rho_{NIR} - \rho_{RED}}{\rho_{NIR} + \rho_{RED}}$$

其中：
- $\rho_{NIR}$ = 近红外波段反射率
- $\rho_{RED}$ = 红光波段反射率
- NDVI值域：-1 ~ +1
- 健康植被典型值：0.2 ~ 0.9
- 裸土/水体：< 0.1

#### 6.2 中国 GB/T 标准 vs 国际标准对比

| 维度 | GB/T 30115-2013(中国) | 国际通用方法 |
|------|----------------------|-------------|
| 标准名称 | 卫星遥感影像植被指数产品规范 | 无统一ISO标准，各机构自定义 |
| 植被指数种类 | NDVI/EVI/RVI/SAVI等 | NDVI(最广泛)/EVI(MODIS官方)/NDWI |
| 级别划分 | 明确1-4级产品级别 | 依数据源(Sentinel/Landsat/MODIS)分级 |
| 产品构成 | 指数栅格+元数据+质量评估报告 | 各地自定义 |
| 质量评价 | 含几何精度+辐射精度+时间一致性 | NASA/ESA 各有内部标准 |
| 空间基准 | 强制CGCS2000 | WGS84或本地坐标系 |
| 参考规范 | GB/T 41280-2022 植被覆盖度产品规范、NY/T 4065-2021 农作物物候期遥感监测 | - |

#### 6.3 本项目植被指数实现

| 植被指数 | 公式 | 用途 | 标准依据 |
|---------|------|------|---------|
| NDVI | (NIR-RED)/(NIR+RED) | 通用植被生长状态 | GB/T 30115-2013 |
| EVI | 2.5×(NIR-RED)/(NIR+6×RED-7.5×BLUE+1) | 高植被覆盖区（抗饱和） | MODIS EVI |
| NDWI | (GREEN-NIR)/(GREEN+NIR) | 水体提取 | McFeeters 1996 |
| SAVI | (NIR-RED)/(NIR+RED+L)×(1+L), L=0.5 | 低植被覆盖区（土壤校正） | Huete 1988 |
| VARI | (GREEN-RED)/(GREEN+RED-BLUE) | RGB相机植被评估 | Gitelson 2002 |

### 七、正射影像(DOM)精度标准——CH/T vs ASPRS 国际对比

#### 7.1 中国测绘行业标准(DOM)

**依据标准**：
- CH/T 9009.3-2010 基础地理信息数字成果 1:5000~1:100000 数字正射影像图

#### 9.2 本项目土方量计算公式

**数字高程模型(DEM)法**——网格法体积计算：

$$V = \sum_{i=1}^{n}\sum_{j=1}^{m} \Delta x \cdot \Delta y \cdot (H_{ref}(i,j) - H_{dem}(i,j))$$

**三角网法**（精度更高）：
$$V = \sum_{k=1}^{t} \frac{S_k}{3} \cdot (h_{k1} + h_{k2} + h_{k3})$$

其中Sₖ为第k个三角形在水平面上的投影面积，hₖ₁、hₖ₂、hₖ₃为三个顶点的高差。

**精度等级**（依据GB/T 50353《建筑工程建筑面积计算规范》相关条款）：

| 比例尺 | 格网间距(m) | 体积相对误差 | 适用场景 |
|--------|-----------|-------------|---------|
| 1:500 | 1~5 | ≤3% | 施工土方、基坑 |
| 1:1000 | 5~10 | ≤5% | 场地平整、道路 |
| 1:2000 | 10~20 | ≤8% | 区域土方估算 |
